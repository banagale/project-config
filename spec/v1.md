# project-config.yaml Specification v1

**Status:** Draft
**Version:** 1
**Last updated:** 2026-02-14

---

## 1. Introduction

`project-config.yaml` is a configuration file that lives at the root of a software project. It provides a single source of truth for project settings consumed by development tools: CLI utilities, sync scripts, terminal integrations, AI agent frameworks, and task management systems.

The format covers the following categories:

- **Project identity** - canonical name and description
- **Git workflow** - branch conventions and remote configuration
- **Sync behavior** - how worktrees stay current with the canonical branch
- **Lifecycle hooks** - scripts triggered at specific points in sync and init flows
- **Task management** - integration with project and issue trackers
- **Cross-project AI artifact availability** - sharing skills and agents across projects
- **Terminal integration** - iTerm2 tab coloring and visual cues
- **Worktree management** - registry of git worktrees and their metadata

The goal is to replace scattered, implicit project conventions with a single declarative file that any tool (or contributor, human or AI) can read to understand how a project is structured and operated.

### 1.1 Consumer Tool Ecosystem

This specification is consumed by a variety of tools. The field reference (Section 4) names specific tools as illustrative examples of each consumer category. Common consumer categories include:

- **Sync tool** (e.g., `wt-sync`) - In multi-worktree projects, developers work across several git worktrees simultaneously. A sync tool keeps each worktree's landing branch up to date with the canonical branch by reading the sync strategy, push settings, and hook configuration from `project-config.yaml`. Without a sync tool, developers must manually pull, merge or rebase, and push across every worktree, which is error-prone and tedious in projects with many worktrees.
- **Init tool** (e.g., `wt-init`) - Creates new worktrees and runs post-init hooks (dependency installation, terminal color assignment, IDE setup) so new worktrees are ready to use immediately.
- **Reset tool** (e.g., `wt-reset`) - Returns worktrees and their nested repositories to their landing branches after completing work on an issue.

These tools are not part of this repository. They are external consumers that read `project-config.yaml` to drive their behavior. Any tool that follows this specification can serve the same role.


## 2. File Location and Format

- The file MUST be named `project-config.yaml` and placed at the project root (the same directory as `.git/` or the worktree root).
- The file MUST be committed to version control. In worktree-managed projects, the file is committed on the canonical branch and distributed to all worktrees via sync. Edits to the config should be made in the primary worktree and synced out so all worktrees share the same configuration.
- The file MUST use YAML 1.2 syntax.
- The file MUST be encoded as UTF-8.
- All field names MUST use `snake_case`.
- Paths MAY use `~` for home directory expansion. Consumers MUST expand `~` to the current user's home directory.
- Comments (lines beginning with `#`) are permitted and encouraged for documentation.


## 3. Design Principles

### 3.1 Tolerant Reader

Consumers MUST accept configs containing unknown fields without error. This ensures forward compatibility: a config written for a future version of this spec will not break older tools that only understand v1. Consumers SHOULD ignore unknown fields silently.

### 3.2 Incremental Adoption

A project only needs to declare the fields it uses. Every field beyond `schema_version` and `repo` is optional and has a sensible default. The smallest valid config is two fields; the recommended starter config adds `description` and `canonical_branch`. Projects can add sections as needs arise.

### 3.3 No Required Fields Beyond Identity

Only `schema_version` and `repo` are required. All other fields have defaults documented in this specification. Consumers SHOULD use these defaults when a field is absent.


## 4. Field Reference

Each field entry documents the field name (using dot notation for nested fields), its type, default value, description, and which tools are expected to consume it. Dot notation is descriptive only; in YAML, nested fields are expressed as mappings (e.g., `sync.strategy` means `strategy` nested under the `sync` key).


### 4.1 Core Identity

#### `schema_version`

- **Type:** integer
- **Default:** (none, REQUIRED)
- **Description:** Identifies which version of this specification the config targets. Currently `1`. Consumers SHOULD warn when encountering an unknown version but SHOULD NOT reject the config outright.
- **Consumers:** All tools that read `project-config.yaml`.

#### `repo`

- **Type:** string
- **Default:** (none, REQUIRED)
- **Description:** The canonical project identifier. Used as the project name across tools, dashboards, and cross-project references. This SHOULD match the git repository name but is not required to.
- **Consumers:** All tools that read `project-config.yaml`.

#### `description`

- **Type:** string
- **Default:** null
- **Description:** A human-readable, one-line description of the project. Intended for display in dashboards, project listings, and AI context.
- **Consumers:** Task managers, project dashboards, AI agent context.


### 4.2 Git Workflow

#### `canonical_branch`

- **Type:** string
- **Default:** `"main"`
- **Description:** The source-of-truth branch for the project. All worktrees sync against this branch. This is the branch that represents the latest stable state of the project.
- **Consumers:** Sync tools (e.g., `wt-sync`), worktree management scripts.

#### `remote`

- **Type:** string or null
- **Default:** `"origin"`
- **Description:** The git remote name used for push and fetch operations during sync. When set to `null`, the project operates in **local-only mode**: all fetch, push, and remote-tracking operations are skipped. Sync operations reference local branches directly (e.g., `git merge main` instead of `git merge origin/main`). This is useful for projects that cannot have a remote, such as repositories containing sensitive personal data. All worktrees in a local-only project share the same `.git` directory, so commits on one worktree's branch are immediately visible to other worktrees without fetching.
- **Consumers:** Sync tools, worktree management scripts.


### 4.3 Sync Configuration

All sync fields live under the `sync` key.

#### `sync.strategy`

- **Type:** string (enum)
- **Default:** `"merge"`
- **Allowed values:** `"merge"`, `"rebase"`, `"ff-only"`
- **Description:** The git strategy used when syncing a worktree's landing branch with the canonical branch.
  - `"merge"` - Merge the canonical branch into the landing branch.
  - `"rebase"` - Rebase the landing branch onto the canonical branch.
  - `"ff-only"` - Fast-forward only; fail if the landing branch has diverged.
- **Consumers:** Sync tools (e.g., `wt-sync`).

#### `sync.push_landing`

- **Type:** boolean
- **Default:** `true`
- **Description:** Whether to push the landing branch to the remote after a successful sync. When `true`, the remote copy of each landing branch is kept up to date automatically.
- **Consumers:** Sync tools.

#### `sync.push_to_canonical`

- **Type:** boolean
- **Default:** `false`
- **Description:** Whether to push directly to the canonical branch on the remote after sync. Most projects use pull requests instead, so this defaults to `false`. Set to `true` only for projects where direct push to the canonical branch is the intended workflow.
- **Consumers:** Sync tools.


### 4.4 Hooks

All hook fields live under the `hooks` key. Each hook specifies a shell script (or scripts) to run at a specific point in the worktree lifecycle. Every hook accepts `string`, `string[]`, or `null`. When specified as an array, scripts execute in order. A null value means no script is configured.

Hook paths MAY be absolute, relative to the config file (project root), or use `~` for home directory expansion.

#### `hooks.post_init`

- **Type:** string, string[], or null
- **Default:** null
- **Description:** Script or scripts to run after a new worktree is created. This is the most commonly used hook. Common uses include assigning iTerm2 tab colors, installing dependencies, and setting up IDE configuration.
- **Consumers:** Worktree init tools (e.g., `wt-init`).

#### Sync hooks: `pre_sync_all`, `post_sync_all`, `pre_sync`, `post_sync`

- **Type:** string, string[], or null
- **Default:** null
- **Description:** Scripts to run at various points during worktree sync operations. `pre_sync_all` and `post_sync_all` run once per sync-all invocation. `pre_sync` and `post_sync` run once per worktree being synced.
- **Consumers:** Sync tools (e.g., `wt-sync`).

Per-worktree hooks (`pre_sync`, `post_sync`, `post_init`) receive the following environment variables:

| Variable | Description |
|---|---|
| `WORKTREE_NAME` | The `name` field of the worktree being operated on |
| `WORKTREE_PATH` | The filesystem path of the worktree |
| `LANDING_BRANCH` | The landing branch of the worktree |

The batch hooks (`pre_sync_all`, `post_sync_all`) do not receive per-worktree environment variables.


### 4.5 Task Management

All task management fields live under the `task` key. This section configures integration with project and issue tracking systems.

#### `task.type`

- **Type:** string (enum)
- **Default:** `"none"`
- **Allowed values:** `"bloon"`, `"none"`. Reserved for future standardization: `"linear"`, `"github"`, `"jira"`.
- **Description:** The task management system used by this project. When set to `"none"`, no task management integration is active. Values listed as "reserved" are recognized identifiers but do not yet have standardized semantics in this specification; tool support for reserved values varies.
- **Consumers:** Task management CLIs, AI agent context.

#### `task.project`

- **Type:** string
- **Default:** value of `repo`
- **Description:** The project name within the task management system. Defaults to the `repo` value if not specified. Useful when the task system uses a different project name than the git repository.
- **Consumers:** Task management CLIs.

#### `task.default_priority`

- **Type:** integer
- **Default:** `2`
- **Description:** Default priority assigned to newly created tasks. Uses a 0-3 scale where 0 is critical and 3 is low.
  - `0` - Critical
  - `1` - High
  - `2` - Medium
  - `3` - Low
- **Consumers:** Task management CLIs.

#### `task.autofocus`

- **Type:** boolean
- **Default:** `true`
- **Description:** Whether to automatically set focus to a newly created task. When `true`, creating a task also makes it the active/focused task in the task management system.
- **Consumers:** Task management CLIs.

#### `task.require_validation`

- **Type:** boolean
- **Default:** `false`
- **Description:** Whether validation is required before marking a task as done. When `true`, tasks must pass a validation step (defined by the task management system) before they can be completed.
- **Consumers:** Task management CLIs.

#### `task.tags_prefix`

- **Type:** string or null
- **Default:** null
- **Description:** A prefix applied to auto-generated tags. When set, tags created by tooling are prefixed with this value to namespace them within the task system.
- **Consumers:** Task management CLIs.

#### `task.external`

- **Type:** object or null
- **Default:** null
- **Description:** Configuration for external project management system integrations. Each key under `external` is a system identifier containing system-specific settings. These keys are defined for forward compatibility; semantics and tool support may vary while the corresponding `task.type` identifiers remain reserved.

##### `task.external.linear`

Configuration for Linear integration.

- **`task.external.linear.team`** (string) - The Linear team prefix (e.g., `"ACME"`). Used to scope operations to a specific team.
- **`task.external.linear.skill`** (string) - The name of the AI skill used for Linear operations.
- **`task.external.linear.user_id`** (string) - The user's ID in Linear. Used for assignment and filtering.

##### `task.external.github`

Configuration for GitHub Issues integration.

- **`task.external.github.repo`** (string) - The GitHub repository in `"owner/repo"` format.
- **`task.external.github.skill`** (string) - The name of the AI skill used for GitHub issue operations.


### 4.6 Cross-Project Availability

All cross-project availability fields live under the `cross_project_availability` key. This section controls which AI artifacts (skills, agents) from this project are made available across all of the user's projects.

Artifacts are discovered from `skills/` and `agents/` directories at the repository root. Common alternate locations (`<project_root>/.claude/skills/`, `<project_root>/.claude/agents/`) are also recognized by implementations.

How an implementation identifies valid artifacts within these directories is not prescribed by this specification. Some implementations treat a directory as a skill only if it contains `SKILL.md`, or as an agent only if it contains `AGENT.md`.

When multiple projects export an artifact with the same name, a collision occurs. Consumers SHOULD warn on collisions and SHOULD behave deterministically (e.g., by using a stable scan order so the first project scanned wins).

#### `cross_project_availability.skills`

- **Type:** string or string[]
- **Default:** `"none"`
- **Allowed values (when string):** `"all"`, `"none"`
- **Description:** Controls which skills from this project's `skills/` directory are available to other projects.
  - `"all"` - Every skill in `skills/` is shared across projects.
  - `"none"` - No skills are shared.
  - When specified as an array of strings, each entry is the name of a specific skill to share.
- **Implementation note:** A reference implementation (`scripts/sync-cross-project.sh`) scans all project configs and manages symlinks in `~/.claude/skills/` based on this field. Run with `--dry-run` to preview changes.
- **Consumers:** AI agent frameworks, skill discovery tools.

#### `cross_project_availability.agents`

- **Type:** string or string[]
- **Default:** `"none"`
- **Allowed values (when string):** `"all"`, `"none"`
- **Description:** Controls which agents from this project's `agents/` directory are available to other projects. Accepts the same values as `cross_project_availability.skills`.
- **Consumers:** AI agent frameworks, agent discovery tools.


### 4.7 Terminal Integration

#### `color_palette`

- **Type:** string[] or null
- **Default:** null
- **Description:** An array of hex color strings (e.g., `"#F97316"`) used for iTerm2 tab coloring. Colors are assigned to worktrees in the order they appear in the `worktrees` array. When null, the consuming tool's built-in default palette is used. The palette SHOULD contain at least as many colors as the project has active (non-archived) worktrees. **Precedence:** If a worktree entry specifies `meta.color`, that value takes priority over the palette assignment for that worktree.
- **Consumers:** iTerm2 tab color scripts (e.g., `post-init-color.sh`, `iterm-tab-worktree-setup.sh`).

#### `alias_prefix`

- **Type:** string or null
- **Default:** null
- **Description:** A short prefix used for generating shell aliases for worktree navigation. When set, worktree init tools can suggest aliases like `{prefix}1`, `{prefix}2`, etc. for quick `cd` to worktree directories. Values SHOULD match `[A-Za-z_][A-Za-z0-9_]*` and SHOULD NOT contain whitespace.
- **Consumers:** Worktree init tools (e.g., `wt-init`).

#### `default_app`

- **Type:** string or null
- **Default:** null
- **Description:** The default application or service to launch when a project-level "run" or "start" command is invoked without arguments. The interpretation of this value is project-specific. For example, a monorepo with multiple apps might set this to the app that developers work on most frequently. Consumers MUST ignore unknown values and MUST NOT fail on values they do not recognize.
- **Consumers:** Dev server scripts, project-level run commands.

#### `extra_windows`

- **Type:** array of objects or null
- **Default:** null
- **Description:** Configuration for additional terminal windows beyond the primary worktree tabs. Each entry specifies a window title and which worktrees it should contain. This is used by terminal restoration scripts that rebuild a multi-window development layout.

Each object in the array supports:
- **`title`** (string, REQUIRED) - The window title. MUST be a non-empty string.
- **`worktrees`** (string[], REQUIRED) - List of worktree names to open in this window. MUST be a non-empty array. Each entry MUST match a declared `worktrees[].name` value. Duplicate entries SHOULD NOT appear.

- **Consumers:** Terminal window restoration scripts (e.g., `iterm-restore-windows.sh`).


### 4.8 Worktrees

#### `worktrees`

- **Type:** array of objects
- **Default:** `[]`
- **Description:** Registry of git worktrees for the project. Each entry represents a worktree checkout. The first entry is conventionally the primary worktree (the main checkout of the repository).

Each object in the array supports the following fields:

##### `worktrees[].name`

- **Type:** string
- **Default:** (none, REQUIRED within a worktree entry)
- **Description:** A short identifier for the worktree. Used in display, hook environment variables, and as a reference in commands. Common conventions: `"primary"` for the main checkout, `"wb1"`, `"wb2"`, etc. for additional worktrees, or descriptive names like `"liquid-glass"` for feature-specific worktrees.
- **Consumers:** Sync tools, worktree management scripts, terminal integration.

##### `worktrees[].path`

- **Type:** string
- **Default:** (none, REQUIRED within a worktree entry)
- **Description:** Filesystem path to the worktree directory. MAY be absolute, use `~` for home directory expansion, or be relative to the config file location.
- **Consumers:** Sync tools, worktree management scripts, terminal integration.

##### `worktrees[].landing_branch`

- **Type:** string
- **Default:** (none, REQUIRED within a worktree entry)
- **Description:** The branch this worktree stays checked out on. During sync, this branch is updated from the canonical branch using the configured sync strategy.
- **Consumers:** Sync tools, worktree management scripts.

##### `worktrees[].archived`

- **Type:** boolean
- **Default:** `false`
- **Description:** When `true`, this worktree is excluded from sync-all and status operations. Archived worktrees remain in the registry for reference but are not actively managed. Useful for parking feature branches or decommissioned worktrees without losing their configuration history.
- **Implementation note:** Consumers SHOULD respect this field when iterating worktrees for batch operations. Support may vary across tools; some may filter archived worktrees from sync and status operations while others may not.
- **Consumers:** Sync tools, status dashboards.

##### `worktrees[].meta`

- **Type:** object
- **Default:** `{}`
- **Description:** Arbitrary metadata attached to the worktree. This field is not read by core tooling and exists as an extension point. Common uses include:
  - `color` (string) - Hex color for iTerm2 tab coloring (e.g., `"#F97316"`).
  - `role` (string) - Worktree role designation (e.g., `"orchestrator"`, `"worker"`).
  - `ports` (object) - Per-worktree port assignments for dev servers (e.g., `{ app: 4001, core: 4101 }`). Enables port isolation when running multiple worktrees simultaneously.
  - `archived_date` (string) - ISO 8601 date when the worktree was archived.
  - `note` or `notes` (string) - Free-text notes about the worktree's purpose or status.
  - `stash` (string) - Description of stashed work in the worktree.
- **Consumers:** Tool-specific; not read by core sync/init tooling.

##### `worktrees[].nested_repos`

- **Type:** array of objects
- **Default:** `[]`
- **Description:** Nested git repositories within the worktree. Some projects contain sub-repositories (not submodules) that need independent branch and sync management.

Each object in the array supports the following fields:

- **`worktrees[].nested_repos[].path`** (string, REQUIRED) - Path to the nested repository, relative to the worktree root.
- **`worktrees[].nested_repos[].landing_branch`** (string, REQUIRED) - The branch the nested repo stays on.
- **`worktrees[].nested_repos[].remote`** (string, default: `"origin"`) - The git remote name for the nested repo.

The `nested_repos` registry allows tools to discover and manage sub-repositories without hardcoding paths. For example, a sync tool can iterate nested repos to reset them to their landing branches, or a status tool can report their state alongside the parent worktree. Consuming tools determine their own sync strategy for nested repos (reset, rebase, merge) based on the project's workflow.



## 5. Minimal Valid Config

The smallest valid `project-config.yaml` requires only two fields:

```yaml
schema_version: 1
repo: my-project
```

All other fields assume their documented defaults. This is sufficient for tools to identify the project and apply default behavior.

A slightly more common minimal config includes a description and explicit canonical branch:

```yaml
schema_version: 1
repo: my-project
description: A short description of the project
canonical_branch: main
```


## 6. Versioning

- The `schema_version` field identifies which version of this specification the config targets.
- Version 1 is the current and only version.
- Future versions will be published as separate specification documents (e.g., `spec/v2.md`) and will include migration guidance from prior versions.
- Consumers encountering an unknown `schema_version` SHOULD emit a warning but SHOULD NOT reject the config. This allows older tools to partially process configs written for newer spec versions, consistent with the tolerant reader principle (Section 3.1).
- New fields MAY be added to v1 in a backward-compatible manner (with defaults that preserve existing behavior). Breaking changes require a new major version.


## 7. Examples

The `examples/` directory in this repository contains `project-config.yaml` files drawn from real projects. These demonstrate common patterns including:

- Minimal configs with only identity fields
- Multi-worktree setups with color palettes
- Task management integration
- Cross-project skill sharing
- Post-init hook chains for worktree bootstrapping
- Archived worktrees with metadata

Consult the examples for practical usage patterns beyond what this specification covers.


## 8. Security Considerations

- **Hooks execute arbitrary code.** The `hooks` fields specify shell scripts that run with the user's permissions. Consumers SHOULD NOT execute hooks from untrusted configs without user confirmation. Projects that accept `project-config.yaml` contributions via pull request SHOULD review hook changes carefully.
- **Cross-project availability exposes artifacts.** When `cross_project_availability.skills` or `cross_project_availability.agents` is set to `"all"`, every project on the machine can access those artifacts. Users should review what is being shared, especially in multi-tenant or shared-machine environments.
- **Path expansion.** Consumers that expand `~` or resolve relative paths SHOULD validate that the resulting path is within expected boundaries. Symlink traversal and path injection are potential risks if configs are consumed from untrusted sources.

## Appendix B: Compatibility Notes

### Field Name Migration

Early drafts of this specification used `publish_to_canonical` as an alternative name for this field. Consumers MAY accept `publish_to_canonical` as an alias for `sync.push_to_canonical` for compatibility. New configs SHOULD use `sync.push_to_canonical`.
